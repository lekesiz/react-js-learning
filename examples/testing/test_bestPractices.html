Testing React — a short cheat sheet

A compressed summary of what holds up in real codebases. These aren't laws — treat them as defaults you bend with reason.

What to test

Test behaviour, not implementation "When the user clicks Save, the handler receives the form values." Not "setState was called with X."
Test at the component boundary Render the component, drive it with public props + user events, assert on the DOM output. You can refactor internals freely.
Cover the three branches of async Loading, success, error. Each is a separate test case.

What NOT to test

Don't assert on CSS class names Brittle and low-signal. Prefer data-testid, aria-*, or visible text.
Don't test library internals No need to verify that useState stores values. Test your component.
Don't rely on sleeps Use waitFor, findBy*. Arbitrary setTimeout waits flake under load.

Query priority (Testing Library)

1. getByRole("button", { name: /save/i })   // most accessible
2. getByLabelText("Email")                   // for form inputs
3. getByText("Welcome")                      // for visible text
4. getByTestId("confirmation")               // last resort

Pattern: Arrange · Act · Assert

it("saves the form", () => {
  // Arrange
  const onSave = vi.fn();
  render(<Form onSubmit={onSave} />);

  // Act
  fireEvent.change(screen.getByLabelText("Name"), { target: { value: "Ada" } });
  fireEvent.click(screen.getByRole("button", { name: /save/i }));

  // Assert
  expect(onSave).toHaveBeenCalledWith({ name: "Ada" });
});

Mocking

Prefer faking over deep mocking A small in-memory fake of your API keeps tests readable. vi.mock("./api") is fine but drifts from real behaviour over time.
Stub at the network boundary when possible Tools like msw intercept fetch and let your component code run untouched — tests match production closely.

Speed

Unit tests: < 100 ms each A thousand of them finish in under a minute. Any slower and people stop running the suite locally.
